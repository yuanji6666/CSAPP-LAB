# lab4 可重定位目标文件、可执行文件与动态链接分析

## 任务2 可重定位目标文件解析

### 1. 节头表
使用 `readelf -S -W main.o` 可以查看目标文件的节头表。主要列含义如下：

- Nr：节编号。
- Name：节名。
- Type：节类型，表示该节存放的数据类别。
- Address：节装入内存后的地址。对可重定位目标文件来说通常为 0。
- Off：该节在文件中的偏移。
- Size：节大小。
- ES：节表项大小，常用于符号表、重定位表等“表格型”节。
- Flg：节标志，表示读、写、执行等属性。
- Lk：链接信息字段，通常指向相关节的索引。
- Inf：附加信息字段，不同节类型含义不同。
- Al：对齐要求。

### 2. 与全局变量及静态局部变量相关的节

#### 2.1 .data 节
`.data` 用来保存已初始化的全局变量和静态变量。

在 main.s 中：

- `init_gv1` 定义为 `10`，所以放入 `.data`。
- `init_gstr1` 定义为字符串 `"U202414862"`，所以放入 `.data`。
- `init_garr1` 定义为 `11, 12, 13`，所以放入 `.data`。
- `gv` 定义为 `6`，所以也放入 `.data`。

`readelf -x .data main.o` 的十六进制内容与初始化值对应关系如下：

- `0x00000000 0a000000` 对应 `init_gv1 = 10`。
- `0x00000004 00000000` 是为了对齐产生的填充字节。
- `0x00000008 55323032 34313438 36320000` 对应 `init_gstr1 = "U202414862"`，最后的 `00` 是字符串结束符。
- `0x00000018 0b000000` 对应 `init_garr1[0] = 11`。
- `0x0000001c 0c000000` 对应 `init_garr1[1] = 12`。
- `0x00000020 0d000000` 对应 `init_garr1[2] = 13`。
- `0x00000024 06000000` 对应 `gv = 6`。

这里体现了两个特点：

- 整数按小端序存储。
- 字符串按字符字节序存储，并以 0 结尾。
- 变量之间可能因 `.align` 产生补零。

#### 2.2 .bss 节
`.bss` 用来保存未初始化变量，或者初始化为 0 的全局/静态变量。它在目标文件里通常只记录大小，不存实际初值。

在 main.s 和后续位置中：

- `uninit_gv2` 没有显式初值，所以进 `.bss`。
- `uninit_gstr2` 没有显式初值，所以进 `.bss`。
- `uninit_garr2` 没有显式初值，所以进 `.bss`。
- `uninit_gp2` 没有显式初值，所以进 `.bss`。
- `uninit_gfp2` 没有显式初值，所以进 `.bss`。

原因很简单：这些对象在程序装入时只需要被清零，不需要在文件中保存初始内容，因此用 `.zero` 或在 `.bss` 中占位即可。

#### 2.3 .data.rel.local 和 .data.rel
这两类节保存“带地址的初始化数据”，也就是初始化值本身要到链接阶段才能最终确定的对象。

在 main.s 附近：

- `init_gp1 = &init_gv1`，放在 `.data.rel.local`。
- `init_gp2 = &uninit_gv2`，也放在 `.data.rel.local`。
- `init_gfp1 = fadd`，放在 `.data.rel`。

一般理解为：

- `.data.rel.local`：指向本对象内部或局部可解析符号的初始化地址。
- `.data.rel`：指向可能需要更一般重定位处理的地址，例如函数地址。

#### 2.4 .strtab 节
`.strtab` 是符号名字符串表，里面保存符号的名称，例如：

- 全局变量名
- 静态局部变量名
- 函数名
- 局部临时符号名
- 未定义符号名

例如 `init_gv1`、`uninit_gv2`、`main`、`fcount`、`fsub`、`count.0`、`printf`、`fadd` 等名字都会通过字符串表被引用。

#### 2.5 一个全局变量定义语句在目标文件中的信息分布
以 `int init_gv1 = 10;` 为例，它在可重定位目标文件中通常会生成：

- `.data` 中的实际初始值内容。
- 符号表中的符号项，记录名字、大小、所在节和节内偏移。
- 如果这个变量被其他地方引用，还会在相关节产生重定位信息。

如果是 `int uninit_gv2;`，则通常会生成：

- `.bss` 中的占位大小。
- 符号表中的符号项。
- 因为没有实际初值，所以不在 `.data` 中存字节内容。

### 3. 代码节及代码节的重定位节

#### 3.1 .text 和 .rela.text
`.text` 保存函数的机器指令。

`.rela.text` 保存代码中的重定位记录，核心信息包括：

- 需要修正的位置。
- 需要引用的符号。
- 重定位类型。
- 加数信息。

这些信息用于告诉链接器：某条指令里有地址或符号引用，现在还不能最终确定，等链接时再补。

在本例中，`main`、`fcount`、`fsub` 都在 `.text` 中。像 `call printf@PLT`、`call fcount`、`call fsub`、`call check_gcc_condition` 这样的指令，通常都与 `.rela.text` 中的重定位记录相关。

#### 3.2 .text 中定义了哪些符号
从 main.s 开始，`.text` 中主要定义了：

- `fcount`
- `fsub`
- `main`

这些都是函数符号。

### 4. 符号表节

#### 4.1 符号表各列含义
`readelf -s main.o` 输出的常见列含义如下：

- Num：符号序号。
- Value：符号值。对可重定位目标文件来说，通常是节内偏移。
- Size：符号大小。
- Type：符号类型，例如 OBJECT、FUNC。
- Bind：符号绑定属性，例如 LOCAL、GLOBAL、WEAK。
- Vis：可见性。
- Ndx：符号所在节索引；如果是 UND，则表示未定义。
- Name：符号名。

#### 4.2 未定义符号
未定义符号是指在当前目标文件中只被引用、但没有定义的符号。常见原因是：

- 函数定义在别的源文件或库中。
- 全局变量定义在别的目标文件中。
- 运行时由系统提供。

在本例里，常见未定义符号包括：

- `printf`
- `fadd`
- `check_gcc_condition`
- `__stack_chk_fail`
- `extern_int`

它们在当前目标文件中只是引用，真正定义在其他目标文件或运行库里。

### 5. 只读节

#### 5.1 .rodata 的内容
`.rodata` 存放只读常量，主要包括：

- 字符串字面量。
- 常量表。
- 某些只读初始化数据。

在本例中，`.rodata` 里有：

- `auto x = %d , static count =%d\n`
- `display init_gstr1 : %s \n`
- `sum = %d \n`

#### 5.2 printf 调用时格式串地址如何定位
以 `printf("display init_gstr1 : %s \n", init_gstr1);` 为例，在 main.s 附近，编译器会先用 `leaq .LC1(%rip), %rax` 取出格式串地址，再把它放入参数寄存器 `%rdi`，然后调用 `printf@PLT`。

这说明：

- 格式串本身在 `.rodata` 中。
- 调用时通过 RIP 相对寻址定位其首地址。
- 格式串地址作为第一个参数传给 `printf`。

## 任务3 可执行目标文件解析

### 1. 比较 main_sub 和 sub_main 的符号排列顺序
链接顺序会影响最终可执行文件中来自不同目标文件的符号布局。

一般规律是：

- 先出现的目标文件，其全局变量和函数更可能排在前面。
- 后出现的目标文件，其符号通常排在后面。
- 同一个目标文件内部的符号相对顺序通常保持不变。

因此：

- `gcc ... main.c sub.c -o main_sub` 中，来自 main.c 的符号通常先于 sub.c。
- `gcc ... sub.c main.c -o sub_main` 中，来自 sub.c 的符号通常先于 main.c。

这说明可执行文件中的地址布局不仅取决于源码，还受链接输入顺序影响。

### 2. 可执行文件符号表与可重定位目标文件符号表的差异
在可重定位目标文件中：

- `Value` 通常是节内偏移。
- 地址尚未最终确定。

在可执行文件中：

- `Value` 变成了运行时的最终虚拟地址。
- 很多未定义符号已经被解析成实际地址。
- 与库相关的符号可能进入动态符号表或保持动态链接状态。

因此，同一个符号在两种文件中的 `Value` 含义不同：

- 目标文件里是“相对位置”。
- 可执行文件里是“最终地址”。

### 3. main 函数机器码在目标文件和可执行文件中的差别

在可重定位目标文件中：

- `call` 指令的目标地址通常还未最终确定。
- 对全局变量和外部函数的引用依赖重定位节修正。
- 反汇编看起来更像“待填补的模板”。

在可执行文件中：

- 所有内部引用基本都已落实到最终地址。
- `call`、`lea`、`mov` 等指令里的位移或地址已经被链接器修正。
- 代码中的机器码更接近最终运行状态。

也就是说：

- 目标文件：地址待定。
- 可执行文件：地址已定。

### 4. 基本概念

- 符号定义：某个符号在当前文件中有实体，例如变量存储区或函数代码。
- 符号绑定：符号的作用域属性，常见有 LOCAL、GLOBAL、WEAK。
- 符号引用：代码或数据中使用了某个符号，但符号定义不在当前文件中。
- 重定位：链接器或加载器根据最终地址修正代码或数据中的地址引用。

### 5. 生成可执行程序的基本过程

- 预处理：展开宏、处理条件编译、包含头文件。
- 编译：把 C 代码转成汇编。
- 汇编：把汇编转成目标文件。
- 链接：合并多个目标文件和库，解析符号并完成重定位，生成可执行文件。

## 任务4 静态链接与动态链接生成的可执行程序差异

### 静态链接
静态链接会把所需库代码直接拷贝进可执行文件。

特点如下：

- 可执行文件更大。
- 运行时依赖更少。
- 启动后不需要再找外部共享库中的函数实现。
- 同一库代码会在多个程序中重复存在，内存利用率较低。

### 动态链接
动态链接在运行时由动态链接器加载共享库。

特点如下：

- 可执行文件更小。
- 多个程序可以共享同一份库代码。
- 程序启动时或首次调用时需要解析共享库符号。
- 依赖运行环境中对应的共享库存在。

### 两者差异总结
- 静态链接强调“打包进程序”。
- 动态链接强调“运行时共享与解析”。
- 静态链接更独立，动态链接更灵活。

## 任务5 动态链接库中的函数定位方法

以 `printf` 为例，程序在汇编中通常会看到 `printf@PLT`。

其定位过程是：

- 编译器把对 `printf` 的调用变成对 PLT 表项的调用。
- PLT 表项再通过 GOT 中的入口间接跳转。
- 第一次调用时，GOT 项通常指向动态链接器的解析入口。
- 动态链接器找到 libc 中真正的 `printf` 地址后，把结果写回 GOT。
- 之后再调用时，程序就能直接跳到真正的 `printf` 实现。

这就是典型的延迟绑定机制。

## 结论

通过对 main.o、main.s 和最终可执行文件的分析可以看到：

- 已初始化全局变量主要进入 .data。
- 未初始化全局变量主要进入 .bss。
- 字符串常量进入 .rodata。
- 地址型初始化数据进入 .data.rel.local 或 .data.rel。
- 函数调用和外部符号引用依赖重定位。
- 可执行文件中的最终地址会受到链接顺序影响。
- 动态链接通过 PLT/GOT 完成函数定位。
